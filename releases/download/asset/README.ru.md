## Концепция

ElasticRelay — это бесшовный гетерогенный синхронизатор данных, предназначенный для захвата изменений данных в реальном времени (CDC) из основных OLTP-баз данных (MySQL, PostgreSQL, MongoDB) в Elasticsearch. Он стремится быть более удобным и надёжным, чем существующие решения, такие как Logstash или Flink.

## 🎉 Особенности версии 1.4.4 - Готовая к продакшену CDC-платформа с движком трансформации

**Три основных источника баз данных + корпоративная трансформация данных:**

| Источник | Статус | Возможности |
|----------|--------|-------------|
| **MySQL** | ✅ Готов | Binlog CDC + Начальная синхронизация + Параллельные снимки |
| **PostgreSQL** | ✅ Закалён для продакшена | Логическая репликация + Парсинг WAL + Стабильный переход от снимка к CDC |
| **MongoDB** | ✅ Готов | Change Streams + Шардированные кластеры + Resume-токены |
| **Движок трансформации** | ✅ Готов | Маппинг полей + Маскирование данных + Преобразование типов + Движок выражений |

## Ключевые возможности

- **Мультиисточниковый CDC**: Полная поддержка MySQL, PostgreSQL и MongoDB с захватом изменений в реальном времени
- **Движок трансформации**: Трансформация данных корпоративного уровня с маппингом полей, маскированием данных (телефон, удостоверение личности, email, банковская карта), преобразованием типов, вычислением выражений и условной фильтрацией — обработка на скорости 800 000+ опер./сек
- **Конфигурация без кода**: Конфигурация на основе JSON с визардом GUI (в разработке)
- **Динамическая многотабличная индексация**: Автоматическое создание отдельных индексов Elasticsearch для каждой исходной таблицы с настраиваемыми шаблонами именования (например, `elasticrelay-users`, `elasticrelay-orders`)
- **Встроенное управление данными**: Обработка структурирования данных, анонимизации, преобразования типов, нормализации и обогащения
- **Надёжность по умолчанию**: Использование CDC на уровне журнала транзакций, точное контрольное сохранение для возобновления и идемпотентные записи для обеспечения целостности данных
- **Очередь недоставленных сообщений (DLQ)**: Комплексная обработка сбоев с экспоненциальной задержкой повторных попыток и постоянным хранилищем
- **Параллельная обработка**: Продвинутая параллельная обработка снимков со стратегиями разбиения для больших таблиц
- **Централизованное логирование**: Настраиваемые во время выполнения уровни логирования (debug/info/warn/error) с потокобезопасным глобальным управлением

## Технологический стек

- **Плоскость данных (Go)**: Основная логика синхронизации данных написана на Go (1.25.2+) для высокой конкурентности, низкого потребления памяти и простого развёртывания.
- **Плоскость управления и GUI (TypeScript/Next.js)**: Богатый интерактивный UI для конфигурации и мониторинга (в разработке).
- **API (gRPC)**: Внутренняя связь между компонентами обрабатывается через gRPC для высокой производительности с полными реализациями сервисов.
- **Поддержка баз данных**: 
  - **MySQL CDC**: Продвинутый парсинг binlog с синхронизацией в реальном времени (библиотека go-mysql)
  - **PostgreSQL CDC**: Логическая репликация с парсингом WAL, слотами репликации и публикациями, и закалённый для продакшена переход от снимка к CDC
  - **MongoDB CDC**: Change Streams с поддержкой replica set и шардированных кластеров (mongo-driver)
- **Движок трансформации**: Полный конвейер трансформации данных с маппингом полей, преобразованием типов, маскированием данных (4 стратегии, 5 шаблонов), движком выражений (16 встроенных функций) и условной фильтрацией (10 операторов)
- **Интеграция с Elasticsearch**: Официальный Go-клиент Elasticsearch (v8) с поддержкой массовой индексации
- **Конфигурация**: Конфигурация на основе JSON с автоматическим определением формата и миграцией
- **Надёжность**: Комплексная обработка ошибок, система DLQ и управление контрольными точками
- **Логирование**: Централизованная система управления уровнем логирования с настройкой во время выполнения

## Архитектура

Система состоит из нескольких ключевых компонентов:

- **Исходные коннекторы**: Захват изменений из MySQL (binlog), PostgreSQL (логическая репликация) и MongoDB (потоки изменений).
- **Устойчивый буфер**: Асинхронная очередь событий CDC, разделяющая чтение источника и последующую обработку.
- **Движок трансформации**: Пайплайн трансформации корпоративного уровня с маппингом полей, преобразованием типов, маскированием данных, вычислением выражений и условной фильтрацией.
- **Писатель ES Sink**: Запись данных в Elasticsearch эффективными пакетами с автоматическим управлением индексами.
- **Оркестратор**: Управление жизненным циклом задач синхронизации с поддержкой как устаревшей одноисточниковой, так и многоисточниковой конфигурации.
- **Очередь недоставленных сообщений (DLQ)**: Обработка неудачных событий с экспоненциальной задержкой повторных попыток и постоянным хранилищем.
- **Менеджер контрольных точек**: Постоянное отслеживание позиции (позиции binlog, LSN PostgreSQL, resume-токены MongoDB) для отказоустойчивого возобновления.
- **Плоскость управления**: UI и бэкенд управления конфигурацией (в разработке).

## Быстрый старт

Чтобы быстро запустить ElasticRelay, выполните следующие три простых шага:

### Шаг 1: Сборка
```sh
./scripts/build.sh
```

### Шаг 2: Настройка

#### Настройка MongoDB (требуется для MongoDB CDC)
MongoDB требует режим replica set для Change Streams. Запустите скрипт настройки:
```sh
./scripts/reset-mongodb.sh
```

Или вручную:
```sh
docker-compose down
rm -rf ./data/mongodb/*
docker-compose up -d mongodb
docker-compose up mongodb-init
```

Проверьте готовность MongoDB:
```sh
./scripts/verify-mongodb.sh
```

📚 **Смотрите**: `QUICKSTART.md` для подробных инструкций по настройке MongoDB.

#### Настройка PostgreSQL
Для PostgreSQL убедитесь, что логическая репликация включена:
```sql
-- Включить логическую репликацию в postgresql.conf
wal_level = logical
max_replication_slots = 10
max_wal_senders = 10

-- Создать пользователя с правами репликации
CREATE USER elasticrelay_user WITH LOGIN PASSWORD 'password' REPLICATION;
GRANT CONNECT ON DATABASE your_database TO elasticrelay_user;
GRANT USAGE ON SCHEMA public TO elasticrelay_user;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO elasticrelay_user;
```

#### Файлы конфигурации
Отредактируйте файл конфигурации `./config/parallel_config.json` и убедитесь, что информация о подключении к базе данных и Elasticsearch корректна.

### Шаг 3: Запуск
```sh
./start.sh
```

После выполнения этих шагов ElasticRelay начнёт отслеживать изменения в базе данных и синхронизировать их в Elasticsearch.

## Как запустить

### Требования

- Go (1.25.2+)
- Компилятор Protobuf (`protoc`)
- Elasticsearch (7.x или 8.x)
- **MySQL** (5.7+ или 8.x) с включённым binlog
- **PostgreSQL** (10+ рекомендуется, 9.4+ минимум) с включённой логической репликацией
- **MongoDB** (4.0+) с конфигурацией replica set или шардированного кластера

### Установка

1.  **Установите Go-зависимости и инструменты**:
    ```sh
    go install google.golang.org/protobuf/cmd/protoc-gen-go@v1.28
    go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@v1.2
    ```

2.  **Установите `protoc`**:
    На macOS с Homebrew:
    ```sh
    brew install protobuf
    ```

3.  **Обновите зависимости**:
    ```sh
    go mod tidy
    ```

### Сборка и запуск сервера

#### Быстрая сборка (Разработка)
```sh
# Простая сборка без информации о версии
go build -o elasticrelay ./cmd/elasticrelay

# Запуск сервера
./elasticrelay -config multi_config.json
```

#### Продакшен-сборка (Рекомендуется)
```sh
# Сборка с информацией о версии через Makefile
make build

# Запуск версионированного бинарника
./bin/elasticrelay -config multi_config.json
```

#### Управление версиями
ElasticRelay имеет комплексное управление версиями с инъекцией во время сборки:

```sh
# Просмотр текущей информации о версии с детальными данными сборки
./bin/elasticrelay -version

# Проверка информации о версии через Makefile
make version

# Сборка для разработки (быстрая, без инъекции версии)
make dev

# Продакшен-сборка (оптимизированная с информацией о версии)
make release

# Кросс-платформенная сборка для нескольких архитектур
make build-all

# Сборка с кастомной версией
VERSION="v1.3.0" make build

# Сборка всех инструментов, включая утилиты миграции
make build-tools
```

Система версий включает:
- **Интеграция с Git**: Автоматическое определение версии из git-тегов
- **Метаданные сборки**: Хеш коммита, время сборки, версия Go и информация о платформе
- **Цветной вывод**: Богатый консольный вывод с деталями версии и ASCII-арт логотипом
- **Кросс-платформенность**: Поддержка Linux, macOS (Intel/ARM) и Windows

Сервер запустится и будет слушать на порту `50051` по умолчанию.

**Альтернатива**: Вы также можете запустить напрямую без сборки:
```sh
go run ./cmd/elasticrelay -config multi_config.json
```

### Многотабличная конфигурация

ElasticRelay поддерживает как устаревший одиночный конфиг, так и современный мульти-конфиг формат с автоматическим определением и миграцией.

#### Современный мульти-конфиг формат (`multi_config.json`):

```json
{
  "version": "3.0",
  "data_sources": [
    {
      "id": "mysql-main",
      "type": "mysql",
      "host": "localhost",
      "port": 3306,
      "user": "elastic_user",
      "password": "password",
      "database": "elasticrelay",
      "server_id": 100,
      "table_filters": ["users", "orders", "products"]
    },
    {
      "id": "postgresql-main",
      "type": "postgresql",
      "host": "localhost",
      "port": 5432,
      "user": "elastic_user",
      "password": "password",
      "database": "elasticrelay",
      "table_filters": ["users", "orders", "products"],
      "options": {
        "ssl_mode": "disable",
        "slot_name": "elasticrelay_slot",
        "publication_name": "elasticrelay_publication",
        "batch_size": 1000,
        "max_connections": 10,
        "parallel_snapshots": true
      }
    },
    {
      "id": "mongodb-main",
      "type": "mongodb",
      "host": "localhost",
      "port": 27017,
      "user": "elasticrelay_user",
      "password": "password",
      "database": "elasticrelay",
      "table_filters": ["users", "orders", "products"],
      "options": {
        "auth_source": "admin",
        "replica_set": "rs0"
      }
    }
  ],
  "sinks": [
    {
      "id": "es-main",
      "type": "elasticsearch",
      "addresses": ["http://localhost:9200"],
      "options": {
        "index_prefix": "elasticrelay"
      }
    }
  ],
  "jobs": [],
  "global": {
    "log_level": "info",
    "grpc_port": 50051,
    "dlq_config": {
      "enabled": true,
      "storage_path": "dlq",
      "max_retries": 3,
      "retry_delay": "30s"
    }
  }
}
```

#### Устаревший формат конфигурации (`config.json`):

```json
{
  "db_host": "localhost",
  "db_port": 3306,
  "db_user": "elastic_user",
  "db_password": "password",
  "db_name": "elasticrelay",
  "server_id": 100,
  "table_filters": ["users", "orders", "products"],
  "es_addresses": ["http://localhost:9200"]
}
```

Система автоматически определяет формат конфигурации и поддерживает миграцию между форматами. При этом создаются отдельные индексы:
- `elasticrelay-users` для таблицы `users`
- `elasticrelay-orders` для таблицы `orders`  
- `elasticrelay-products` для таблицы `products`

### Поддержка очереди недоставленных сообщений (DLQ)

ElasticRelay включает комплексную систему DLQ для обработки неудачных событий:

- **Автоматический повтор**: Неудачные события автоматически повторяются с экспоненциальной задержкой
- **Постоянное хранилище**: Элементы DLQ сохраняются на диск с полным управлением состоянием
- **Дедупликация**: Предотвращает добавление дубликатов событий в очередь
- **Отслеживание статуса**: Полное отслеживание жизненного цикла (ожидание, повторная попытка, исчерпано, разрешено, отброшено)
- **Ручное управление**: Поддержка ручной проверки и управления элементами
- **Автоматическая очистка**: Разрешённые элементы автоматически очищаются после настраиваемой продолжительности

### Поддержка PostgreSQL

ElasticRelay предоставляет комплексные возможности PostgreSQL CDC с продвинутыми функциями:

#### Основные возможности PostgreSQL
- **Логическая репликация**: Использует нативную логическую репликацию PostgreSQL с плагином `pgoutput`
- **Парсинг WAL**: Продвинутый парсинг Write-Ahead Log для захвата изменений в реальном времени
- **Слоты репликации**: Автоматическое создание и управление слотами логической репликации
- **Публикации**: Динамическое управление публикациями для фильтрации таблиц
- **Управление LSN**: Точное отслеживание Log Sequence Number для функции контрольной точки/возобновления

#### Продвинутые возможности PostgreSQL
- **Пул соединений**: Интеллектуальное управление пулом соединений с настраиваемыми лимитами
- **Параллельные снимки**: Многопоточная начальная синхронизация данных со стратегиями разбиения
- **Маппинг типов**: Комплексное преобразование типов PostgreSQL в Elasticsearch, включая:
  - Все числовые типы (bigint, integer, real, double, numeric)
  - Текстовые и символьные типы (text, varchar, char)
  - Типы даты/времени с поддержкой часового пояса (timestamp, timestamptz, date, time)
  - JSON/JSONB с нативным маппингом объектов
  - Типы массивов (массивы целых чисел, текстовые массивы)
  - Продвинутые типы (UUID, bytea, inet, геометрические типы)
- **Оптимизация производительности**: 
  - Адаптивное планирование для больших таблиц
  - Потоковый режим для эффективности памяти
  - Настраиваемые размеры пакетов и пулы воркеров
  - Управление жизненным циклом соединений

#### Параметры конфигурации PostgreSQL
```json
{
  "type": "postgresql",
  "options": {
    "ssl_mode": "disable|require|verify-ca|verify-full",
    "slot_name": "custom_replication_slot_name",
    "publication_name": "custom_publication_name",
    "batch_size": 1000,
    "max_connections": 10,
    "min_connections": 2,
    "parallel_snapshots": true,
    "enable_performance_monitoring": true
  }
}
```

### Поддержка MongoDB

ElasticRelay предоставляет полные возможности MongoDB CDC с использованием Change Streams:

#### Основные возможности MongoDB
- **Change Streams**: CDC в реальном времени с использованием нативного API Change Streams MongoDB
- **Поддержка кластеров**: Автоматическое определение и поддержка replica set и шардированных кластеров
- **Resume-токены**: Постоянное управление resume-токенами для функции контрольной точки/возобновления
- **Маппинг операций**: Полная поддержка операций INSERT, UPDATE, REPLACE и DELETE

#### Продвинутые возможности MongoDB
- **Поддержка шардированных кластеров**: 
  - Мониторинг нескольких шардов через mongos
  - Осведомлённость о миграции для согласованности во время миграции чанков
  - Мониторинг распределения чанков
- **Преобразование типов**: Полное преобразование BSON в JSON-совместимые типы:
  - ObjectID → строка (hex-формат)
  - DateTime → метка времени RFC3339
  - Decimal128 → строка (точность сохраняется)
  - Binary → base64 кодировка
  - Вложенные документы с настраиваемой глубиной выравнивания
- **Параллельные снимки**: 
  - Разбиение на основе ObjectID для стандартных коллекций
  - Разбиение на основе числового ID для целочисленных первичных ключей
  - Резервный режим Skip/Limit для сложных типов ID

#### Параметры конфигурации MongoDB
```json
{
  "type": "mongodb",
  "host": "localhost",
  "port": 27017,
  "user": "elasticrelay_user",
  "password": "password",
  "database": "your_database",
  "options": {
    "auth_source": "admin",
    "replica_set": "rs0",
    "read_preference": "primaryPreferred",
    "batch_size": 1000,
    "flatten_depth": 3
  }
}
```

#### Требования к настройке MongoDB
```sh
# MongoDB должен работать в режиме replica set для Change Streams
# Используйте предоставленный скрипт настройки:
./scripts/reset-mongodb.sh

# Или с Docker Compose:
docker-compose up -d mongodb
docker-compose up mongodb-init

# Проверьте, что replica set настроен:
./scripts/verify-mongodb.sh
```

### Движок трансформации

ElasticRelay включает полный конвейер трансформации данных, настраиваемый отдельным JSON-файлом (`-transform-config`):

#### Маппинг полей
- **Переименование**: Изменение имён полей (например, `user_name` → `username`)
- **Копирование**: Дублирование полей под новыми именами с сохранением исходных
- **Вложенные пути**: Доступ и изменение вложенных полей через точечную нотацию (`user.profile.name`)
- **Исключение полей**: Удаление чувствительных или ненужных полей до индексации

#### Преобразование типов

| Исходный тип | Целевые типы |
|--------------|--------------|
| string | int, int64, float64, bool, date, timestamp |
| int/int64 | string, float64, bool, timestamp |
| float64 | string, int, int64, bool |
| bool | string, int |
| time.Time | string (RFC3339), timestamp (Unix) |

#### Маскирование данных

| Шаблон | Вход | Выход |
|--------|------|-------|
| `phone` | `13812345678` | `138****5678` |
| `id_card` | `110101199001011234` | `1101**********1234` |
| `email` | `john@example.com` | `jo***@example.com` |
| `bank_card` | `6222021234567890` | `6222********7890` |
| `name` | `张三` | `张*` |

Стратегии маскирования: `mask` (символьное маскирование), `hash` (SHA256/MD5), `token` (токенизация), `regex` (замена по шаблону).

#### Движок выражений

Встроенные функции для вычисляемых полей:

| Категория | Функции |
|-----------|---------|
| Строки | `concat()`, `substr()`, `upper()`, `lower()`, `trim()`, `replace()`, `length()` |
| Математика | `round()`, `abs()`, `floor()`, `ceil()`, `min()`, `max()` |
| Дата | `now()`, `formatDate()`, `parseDate()` |
| Условия | `ifNull()`, `ifEmpty()`, `coalesce()` |

Примеры выражений:
```javascript
$.age < 18 ? 'minor' : 'adult'
concat($.first_name, ' ', $.last_name)
round($.price * $.quantity, 2)
```

#### Условная фильтрация

| Оператор | Описание | Пример |
|----------|----------|--------|
| `eq` | Равно | `status == "active"` |
| `ne` | Не равно | `status != "deleted"` |
| `gt` / `gte` | Больше (или равно) | `age > 18` |
| `lt` / `lte` | Меньше (или равно) | `price < 100` |
| `in` / `nin` | В списке / не в списке | `type in ["a", "b"]` |
| `regex` | Совпадение по regex | `email ~ ".*@example.com"` |
| `exists` | Поле существует | `email exists` |

#### Конфигурация трансформации

```sh
# Запуск с правилами трансформации
./bin/elasticrelay -config multi_config.json -transform-config ./config/mysql_transform.json

# Запуск без трансформации (режим pass-through, по умолчанию)
./bin/elasticrelay -config multi_config.json
```

### Параллельная обработка

Продвинутые возможности параллельной обработки снимков:

- **Стратегии разбиения**: Поддержка разбиения на основе ID, времени и хеша
- **Пулы воркеров**: Настраиваемые размеры пулов воркеров с адаптивным планированием
- **Отслеживание прогресса**: Мониторинг прогресса и статистики в реальном времени
- **Поддержка больших таблиц**: Оптимизированная обработка больших таблиц с интеллектуальным разбиением
- **Потоковый режим**: Эффективная по памяти потоковая обработка для больших наборов данных
- **Обнаружение первичного ключа**: Автоматическое определение столбцов первичного ключа для корректных идентификаторов документов

## Текущий статус

**Текущая версия**: v1.4.4 | **Фаза**: Фаза 2 завершена ✅, Фаза 3 в процессе (Движок трансформации завершён)

Этот проект завершил основную мультиисточниковую CDC-платформу (Фаза 2) и реализовал движок трансформации как первый крупный этап Фазы 3. PostgreSQL CDC закалён для продакшена благодаря значительным исправлениям стабильности.

### ✅ Завершённые функции (v1.4.4)
- **Мультиисточниковый CDC-пайплайн**: 
  - **MySQL CDC**: Полная реализация с синхронизацией в реальном времени на основе binlog и согласованной обработкой даты и времени
  - **PostgreSQL CDC**: Закалённая для продакшена логическая репликация с парсингом WAL, слотами репликации и публикациями, стабильным переходом от снимка к CDC, асинхронным разделением пакетов и управлением слотами репликации в рамках задачи
  - **MongoDB CDC**: Полная реализация Change Streams с поддержкой replica set и шардированных кластеров
- **Движок трансформации** (v1.4.0+):
  - Маппинг полей (переименование, копирование, перемещение) с поддержкой вложенных путей
  - Преобразование типов (string, int, float, bool, date, timestamp, object)
  - Маскирование данных (телефон, удостоверение личности, email, банковская карта, имя) с 4 стратегиями
  - Движок выражений с 16 встроенными функциями
  - Условная фильтрация с 10 операторами и действиями include/exclude/route
  - Сопоставление нескольких правил по приоритету с шаблонами имён таблиц
  - Производительность: 800 000+ опер./сек (в 80 раз выше целевого показателя проектирования)
- **Многотабличная динамическая индексация**: Автоматическое создание и управление индексами Elasticsearch для каждой таблицы с настраиваемым именованием
- **Архитектура gRPC**: Полные определения и реализации сервисов (Connector, Orchestrator, Sink, Transform, Health)
- **Продвинутое управление конфигурацией**: 
  - Система конфигурации для нескольких источников с поддержкой миграции устаревших версий
  - Синхронизация конфигурации и возможности горячей перезагрузки
  - Автоматическое определение формата и инструменты миграции
- **Интеграция с Elasticsearch**: Высокопроизводительная массовая запись с автоматическим управлением индексами и очисткой данных
- **Контрольная точка/Возобновление**: Постоянное отслеживание позиции для отказоустойчивости с автоматическим восстановлением (binlog, LSN, resume-токены)
- **Очередь недоставленных сообщений (DLQ)**: 
  - Комплексная система DLQ с экспоненциальной задержкой повторных попыток (настраиваемое максимальное количество попыток)
  - Постоянное хранилище с дедупликацией и отслеживанием статуса
  - Автоматическая очистка разрешённых элементов
  - Поддержка ручного управления и проверки элементов
- **Параллельная обработка**: 
  - Продвинутая параллельная обработка снимков со стратегиями разбиения
  - Автоматическое обнаружение первичного ключа для корректной генерации идентификаторов документов
  - Настраиваемые пулы воркеров и адаптивное планирование
  - Отслеживание прогресса и сбор статистики
  - Поддержка оптимизации больших таблиц (MySQL, PostgreSQL, MongoDB)
- **Управление версиями**: Полная система инъекции версий с метаданными времени сборки
- **Надёжная обработка ошибок**: Комплексная обработка ошибок с механизмами резервирования
- **Управление уровнем логирования**: Централизованная система логирования (debug/info/warn/error) с настройкой во время выполнения и потокобезопасным глобальным управлением
